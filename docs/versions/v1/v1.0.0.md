This is a major baseline because it creates the first intended backend service structure.
It turns the earlier script-style analysis idea into a FastAPI API with routes, models,
services, tests, and setup documentation.

## Plain-Language Summary

Version `v1.0.0` is the first working backend baseline for the ReqSmell prototype. It can
accept a CSV upload, parse requirement rows, start an analysis run, and return progress
and reports through API endpoints. It runs in mock LLM mode by default, so a new developer
can test it without Anthropic or OpenAI keys. This version is the clean starting point for
all backend `v1.x` improvements.

## First-Time Setup

```powershell
cd Backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
Copy-Item .env.example .env
python -m uvicorn app.main:app --reload --port 8000
```

Open the API docs:

```text
http://127.0.0.1:8000/docs
```

Health check:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

Expected response:

```text
status
------
ok
```

## Frontend Assumption

This backend supports the current React frontend API contract in
`Frontend/src/types/index.ts`. The frontend Vite proxy expects the backend at:

```text
http://localhost:8000
```

The backend allows local frontend origins configured in `.env.example`:

```text
http://localhost:5173
http://127.0.0.1:5173
```

## Known Limitations

| Limitation | Impact |
|---|---|
| Run state is stored only in memory | Restarting the backend clears all active and completed runs |
| Mock mode is heuristic | It is useful for integration testing but not final research results |
| Live LLM mode depends on API keys | Anthropic and OpenAI calls require valid keys in `.env` |
| No authentication | The backend is intended for local prototype use |
| No durable queue | Long-running analysis is managed by in-process background tasks |

## Upgrade Notes

There is no previous backend version to upgrade from. Future `v1.x` versions should keep
the current API contract compatible unless the version document clearly states otherwise.

## Downgrade Notes

Downgrading from a future `v1.x` version to `v1.0.0` will remove any later features such
as persistent storage, raw response logging, export endpoints, or deployment setup.

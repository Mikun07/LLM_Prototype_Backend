This is a backend feature change because it adds explicit HTTP status behavior to the
existing version 1 baseline.

## Plain-Language Summary

Version `v1.1.0` makes the backend clearer about what happened after each request. Upload
and status polling return `200 OK`, analysis creation returns `201 Created`, and missing
live provider configuration returns `503 Service Unavailable`. Existing run polling still
returns the run object so the frontend can show pipeline-level errors.

## What Changed From The Previous Version

| Area | What changed | Why |
|---|---|---|
| Analysis route | `POST /api/analyse` now returns `201 Created` | A new analysis run resource is created |
| Analysis route | Successful run creation sets a `Location` header | The frontend can immediately know where to poll |
| Provider readiness | Missing selected API keys now return `503` before a run starts | Users get a direct configuration error |
| Provider errors | OpenAI `429 insufficient_quota` is normalized into a readable pipeline error | Users see what to fix instead of a raw provider payload |
| Run status | Runs with failed selected pipelines finish with `status: "error"` | The frontend can distinguish failed runs from completed runs |
| Tests | Route tests now cover `201` and `503` behavior | Protects the status-code contract |
| Documentation | README and version index now list the status policy | Makes setup and debugging clearer |

## What Is Included

| Area | Details |
|---|---|
| Status codes | `200`, `201`, `404`, `413`, `422`, and `503` are documented |
| Provider checks | `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` are checked when live mode is enabled |
| Provider error handling | OpenAI quota exhaustion is not retried and is reported as a clear run error |
| Tests | Backend route and LLM-client tests cover status and quota behavior |
